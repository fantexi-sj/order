# 核心铁律（全局强制执行）

代码修改严格遵循：绝不删除项目内任何有效、有用、业务必需代码，仅做增量开发、精准 Bug 修复、指定逻辑调整。要渲染输入的时候一定要先去store里寻找有没有对应的数据值，有就调用，没有就编写然后在调用，然后就是我的api接口大多数都是要先在store里定义，然后在组件里调用。不要直接在组件里编写api接口。

禁止新增无用代码、重复代码、冗余变量、多余逻辑、无效注释、冗余依赖，所有编写内容必须贴合实际业务需求。

禁止主观猜疑问题、禁止预判潜在问题、禁止解决未出现的可能性问题；需求模糊、字段未知、逻辑不明确、业务规则不清楚时，必须主动提问确认，不擅自假设开发。

全程严格绑定固定技术栈：Taro + React + TypeScript + Zustand，禁止私自引入其他状态库、框架、语法、第三方无关依赖。

所有问题必须先精准定位根因、锁定对应目录 / 文件 / 逻辑模块，只针对精准问题做最小范围修改，不大范围重构、不扩写无关逻辑。

深度熟知微信小程序全部原生 API、基础库机制、权限体系、生命周期、缓存机制、路由规则；完全掌握小程序分包、独立分包、分包预加载全套机制。

***

## 一、项目目录结构规范

- 整体采用 Taro 标准目录分层，所有业务源码统一归集在 src 目录下，配置文件统一放置在项目根目录 config 文件夹。
- 统一划分固定业务模块文件夹：接口请求、全局公共组件、业务专属组件、全局常量、自定义通用 Hooks、页面目录、分包页面根目录、Zustand 状态管理、全局样式、TS 类型定义、工具函数。
- 公共资源与业务资源完全隔离，全局通用组件、工具方法、样式变量、请求封装统一抽离公共层，页面只保留自身业务逻辑。
- 点餐业务按模块拆分：用户模块、菜品分类模块、购物车模块、订单模块、支付模块、评价模块、售后模块，各模块代码独立隔离，互不耦合。
- 分包页面统一归集在专属分包根目录内，和主包页面目录严格区分，禁止分包页面、主包页面文件混放。
- 类型文件统一集中管理，全局通用 TS 类型、业务实体类型、接口请求响应类型分开维护，不零散定义在页面与组件内。

***

## 二、微信小程序分包专属规范

- 严格遵守微信小程序体积限制，主包体积严格管控，仅放置核心高频页面，低频、非刚需页面全部划入分包。
- 核心点餐首页、菜单页、购物车、订单列表、个人中心固定放在主包；支付结果、订单评价、售后退款、客服咨询等低频页面全部采用分包加载。
- 合理配置分包预加载规则，在进入订单核心页面时预加载支付分包，提升用户操作体验。
- 严格限制跨包引用规则，分包仅可引用全局公共工具、公共组件、通用样式，禁止跨分包互相引用、禁止分包直接依赖主包业务私有资源。
- 分包路由跳转必须使用完整绝对路径，区分主包 Tab 路由、普通路由、分包路由的不同跳转方式，严格遵循微信小程序路由 API 使用规则。
- 区分普通分包与独立分包使用场景，点餐项目业务场景统一使用普通分包，保证全局登录态、状态数据互通。

***

## 三、编码开发规范（Taro+React+TS）

- 全部采用 React 函数组件 + Hooks 开发，废弃类组件写法，统一组件开发模式。
- 全项目强制使用 TypeScript，杜绝任意模糊类型，所有变量、函数、入参、出参、组件 Props、接口数据全部定义明确类型。
- 语法统一规范，禁用 var，常量统一使用 const、动态变量使用 let，代码缩进、换行、空格统一遵循格式化规范。
- 业务逻辑分层解耦，页面只负责 UI 渲染与事件触发，复杂计算、接口请求、通用逻辑抽离至 Hooks 或工具函数。
- JSX 结构层级精简，避免过度嵌套，保证代码可读性；禁止直接操作原生 DOM，完全依靠组件状态驱动视图更新。
- 微信小程序原生 API 不直接在页面业务中裸写，全部统一二次封装为通用工具方法，统一处理回调、异常、加载状态。
- 页面生命周期严格遵循 Taro 与小程序双端生命周期规则，合理使用 onShow、onHide、useEffect，避免无效重复请求、重复监听。

***

## 四、Zustand 状态管理强制规范

- 全局唯一状态管理方案为 Zustand，禁止混用 Context、临时全局变量、本地缓存替代全局状态。
- 按业务模块拆分独立仓库，用户信息、购物车、菜品数据、订单数据拆分独立 Store，单一仓库只负责单一业务领域。
- 状态更新遵循不可变数据原则，禁止直接修改原始 State，全部通过官方 Action 方法更新数据。
- 异步请求、接口调用禁止写在 Store 内部，状态仓库仅负责数据存储与同步计算逻辑，异步逻辑统一放在 API 层或自定义 Hooks。
- 组件内按需引入状态，精准解构所需数据与方法，禁止全量引入所有状态，减少不必要渲染。
- 购物车、订单总价、选中菜品等核心联动数据，统一在 Store 内部封装计算方法，保证全局数据一致性。

***

## 五、微信小程序 API 应用规范

- 全面掌握小程序授权、登录、手机号获取、定位、相册、摄像头等权限 API，严格遵循微信授权合规规则，非必要不申请敏感权限。
- 网络请求、文件上传、本地缓存、路由跳转、弹窗提示、加载动画、分享、支付等常用 API 全部统一封装，统一错误拦截与状态提示。
- 本地缓存仅用于存储轻量化数据，例如登录标识、基础用户信息，禁止存储大量菜品列表、订单列表等大数据。
- 微信支付流程严格按照官方规范对接，完整处理支付唤起、支付成功、支付失败、取消支付全场景异常。
- 完善所有 API 异常捕获，网络失败、授权拒绝、接口报错、权限关闭等场景，必须搭配友好文字提示，避免页面白屏、卡死。
- 适配微信不同基础库版本，规避低版本 API 兼容问题，保证小程序多机型、多系统正常运行。

***

## 六、文件、变量、命名统一规范

- 文件夹、普通页面文件统一使用小写横线命名法；通用组件、业务组件采用大驼峰命名。
- 样式文件采用模块化隔离写法，避免全局样式污染，样式类名统一使用小写横线命名。
- 变量、方法采用小驼峰命名，语义化命名，做到见名知意；常量数据、枚举状态统一使用大写加下划线命名。
- 组件入参、TS 接口类型统一大驼峰命名，保持类型定义统一风格。
- 方法命名统一动词开头，例如获取数据、新增操作、删除操作、计算逻辑，逻辑用途清晰直观。

***

## 七、样式开发规范

- 全局样式、主题色、字体大小、边距、圆角、阴影统一抽离全局样式变量，全项目统一复用，方便后期整体换肤修改。
- 组件与页面样式全部使用模块化样式隔离，杜绝全局样式冲突。
- 小程序尺寸单位统一使用 rpx，适配微信小程序多端屏幕，禁止混用 px、rem 等不兼容单位。
- 严格控制行内样式使用，所有样式统一通过 class 控制；非特殊紧急场景，禁止使用强制权重样式。
- 菜品列表、购物车列表等长列表做好样式优化与布局优化，避免卡顿、布局错乱。

***

## 八、TypeScript 严格规范

- 杜绝滥用 any 类型，业务数据、接口返回值全部编写精准 TS 类型与泛型约束。
- 接口通用返回结构统一封装全局泛型类型，减少重复类型编写。
- 新增业务功能时，优先补充对应类型定义，再进行业务逻辑开发。
- 严格校验 TS 编译报错，所有开发内容必须保证无类型错误才可生效。

***

## 九、问题处理 & 开发协作规范

- 接收需求或问题后，先完整梳理业务背景、复现步骤、关联文件，精准锁定问题范围，不扩大修改范围。
- 只解决当下明确提出的问题，不附带优化其他无关功能、不私自升级技术方案。
- 遇到业务规则不明确、接口字段缺失、交互逻辑未说明、限制条件未知的情况，第一时间提问，不自行脑补实现。
- 每一处修改都保持代码最小改动原则，保留原有业务逻辑结构，只修补问题节点。
- 禁止残留调试代码、打印日志、测试注释，正式业务代码保持干净整洁。

***

## 十、性能与项目安全规范

- 菜品图片、资源图片做压缩与懒加载处理，控制图片大小，减少小程序加载体积。
- 长菜品列表、订单列表做好渲染优化，避免一次性渲染大量 DOM 造成页面卡顿。
- 前端仅做展示交互，价格计算、订单状态、库存校验、支付校验核心逻辑全部由后端控制，防止前端数据篡改。
- 接口请求做好防抖、节流、重复请求拦截，优化加载体验。
- 敏感用户数据、登录凭证做好安全存储，避免明文存储与泄露风险。

***

## 十一、补充附加约束规范

- 所有复杂业务逻辑、特殊交互、硬性业务规则，必须添加简洁注释，方便后期维护迭代。
- 严格区分环境，开发环境、体验环境、正式环境配置隔离，不混用接口地址。
- 适配小程序上架审核规则，所有功能、文案、权限使用符合微信官方审核要求。
- 新增页面、新增组件必须遵循现有目录与规范，保持项目整体风格统一、结构统一、维护性统一。
